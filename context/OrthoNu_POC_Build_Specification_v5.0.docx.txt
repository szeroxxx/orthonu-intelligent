1. Executive Context
1.1 What This Document Is
This is a complete build specification for a POC prototype of the OrthoNu Clinical Intelligence Platform. The data is fictional, backend services are simulated, but the UI and UX must look and feel professional and real. This document includes an exhaustive visual design system extracted from an approved reference implementation (the JSX component file) so the developer can match the existing look and feel exactly.
1.2 What This Platform Does
1. A patient arrives. The PMS has their procedures, each identified by a CDT code.
2. The OrthoNu platform ingests the patient’s CDT codes (POC: pre-populated mock patients).
3. The platform looks up each CDT code. A single code can map to MULTIPLE products.
4. All matched product recommendations display: name, indication, price, Dentira CTA.
5. The user triggers simulated procurement. Toast confirms.
6. Analytics track protocols fired, revenue, orders.
1.3 Skeleton-First Architecture
FR-05 (Workflow Visualization) is the application’s state machine. Build it first. Every other feature plugs into its step transitions.
1.4 Patient-Driven Flow
Users do NOT manually type CDT codes. Pre-loaded patients carry procedures. User selects a patient, clicks a procedure, and the workflow triggers.

4.3 State Machine Rules
1. Single active state. Always in one step (1–6). Default: Step 1.
2. Forward-only: 1 → 2 → 3 → 4 → 5 → 6. Backward only via RESET.
3. Automatic: Steps 2 → 3 (300ms) and 3 → 4 (500ms). User sees cascade.
4. User-triggered: 1 → 2 (click procedure), 4 → 5 (click CTA), 5 → 6 (API returns).
5. Reset: returns to Step 1. Triggered by next patient, new procedure, no-match, or tab switch.
6. No-match branch: Step 1 stays active with orange node highlight.
7. Event logging: every transition appends to stepHistory array.
4.4 useWorkflow Hook API
1. currentStep (number 1–6). advanceToStep(step, payload). resetWorkflow().
2. setNoMatchState(cdtCode). noMatchActive (boolean). getStepStatus(step).
3. stepHistory (array of { fromStep, toStep, timestamp, payload }).
4.5 Debug Panel + Acceptance Criteria
Build 8 temporary debug buttons (advance to each step, reset, no-match, log history). Dashed border, labeled “DEBUG — Remove before delivery.” Acceptance: all 6 steps animate, reset works, no-match shows orange, event log is correct, no console errors.

FR-01 — Patient Session Queue   [Must Have]
How the Patient Queue Works
* Each demo session begins with a queue of fictional patients drawn from a pool of 30. On each load of the POC, the system randomly selects 10 patients from the pool and displays them in the queue. This creates variety across demo sessions.
* Each patient carries a name, DOB, patient ID, insurance, provider, and 1–3 scheduled procedures (CDT codes drawn from the real OrthoNu mapping schema).
* The user’s interaction: (1) select a patient from the queue, (2) see that patient’s details and scheduled procedures, (3) click a procedure to trigger the workflow. This simulates how a real PMS integration feeds patient and procedure data to the platform.
Visual Design: Patient Queue (Left Panel)

* Patient meta (age, insurance): DM Mono, 11px, color slate.
* Procedure count badge: MonoTag style (DM Mono, 11px, tealPale bg, teal text).
Visual Design: Procedure Selection (Detail Panel)
* When patient selected, show detail above the queue or as expanded section.

* See Section 5.3 for the complete 30-patient roster.
